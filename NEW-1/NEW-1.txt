## How Vector DB + RAG Changes the Architecture

Right now the Migration Contract (MC) solves the **"agents forget what prior agents decided"** problem by passing JSON between steps. With a Vector DB you can go much further — instead of passing a flat JSON file, you build a **living semantic memory** of your entire migration project.

**The 3 biggest wins you get:**

**1. Semantic Code Search replaces raw JSP injection**
Instead of pasting entire JSP files into agent prompts (which burns context and causes truncation on large forms), you embed all your legacy code into the vector DB upfront. Agents then *retrieve only the relevant chunks* — just the fields, business rules, or servlet methods they actually need for that specific task.

**2. RAG-powered Golden Examples**
Every successfully migrated module gets embedded. When a new agent is generating a complex form, it retrieves the 2–3 most similar previously-migrated forms as automatic few-shot examples. This replaces the manual `golden-examples/` folder we set up — it scales automatically as you migrate more modules.

**3. Cross-module pattern reuse**
Your vector DB becomes a reusable pattern library. When Angular generator agent encounters a conditional validation pattern it has seen before, RAG pulls the already-verified implementation instead of generating from scratch. Consistency goes up dramatically, hallucination goes down.
